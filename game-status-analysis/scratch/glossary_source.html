Production workflow

A pull system across two boards. Each team pulls work in when it has capacity — nobody pushes downstream. Every stage shows its owner; tap a role to highlight where it acts; click any stage for its full detail.

01

Pull, not push

When a stage's team has free capacity, it pulls in games from the previous stage whose Definition of Done is met and that satisfy this stage's Definition of Entry. Nobody pushes work forward.

02

PM watches the whole Game board

The PM is a watcher on every Game-board stage, and the assignee on the coordination-heavy ones — To Do, Contract Alignment, Development, Integration QC, Packaging.

03

DoR vs DoD

DoR = "can we start without redoing it?" (input, clarity, readiness). DoD = "is it done & safe to ship?" (output, quality, testing). If it can only be checked after the work, it's DoD.

04

One epic per game

A game = one epic. The epic holds the docs and links the stories; the actual work lives in tasks/subtasks inside each story. Closed when dev is done + all linked.

How the work is structured — Epic › Story › Task

A game is one epic. The epic only holds the source-of-truth docs and links the stories — never the work itself. The work lives in stories (one per vertical / board, each titled with a [TAG] prefix) and, inside each story, in tasks & subtasks.

Keep the epic clean. Its assignee changes per stage to whoever owns it (the PM on coordination stages); it carries the documents, while every vertical's story carries that team's real tasks.

Read it as three levels: the epic is the game (docs + links only), each story is one team's slice titled by its tag — [GE] [FE] [Platform] [BO] [Data] [Math] [DevOps] [QC] [Bug] — and the actual work happens in the tasks / subtasks inside each story. This keeps the epic readable instead of piling every job onto it.
